# Control Room — Workflow Architecture

This document fixes the workflow model for Control Room after the PoC phase.
It is intentionally deterministic at the pipeline level and LLM-driven only
inside individual steps.

## Goals

- Avoid the assignment/role confusion that makes Paperclip unreliable.
- Make every state transition explicit, auditable, and reversible.
- Keep the orchestrator small and hard to fool.
- Let LLMs generate content, but never let them choose the next pipeline step.

## Core principle: code owns the graph, LLM owns the node content

```
┌──────────────────────────────────────────────┐
│              Control Room                     │
│                                               │
│   Epic  ──►  research  ──►  qa_review         │
│                  │            │               │
│                  ▼            ▼               │
│              pm_plan  ──►  engineering        │
│                               │               │
│                               ▼               │
│                         qa_verify             │
│                               │               │
│                               ▼               │
│                        pm_consistency        │
│                               │               │
│                               ▼               │
│                              done             │
│                                               │
│   Transitions: hard-coded in orchestrator.go  │
│   Step output: generated by Hermes agent     │
└──────────────────────────────────────────────┘
```

## Artifact hierarchy

```
workspace/
├── workspace.yaml
├── epics/
│   └── <epic-id>.json
├── tasks/
│   ├── <task-id>.json
│   └── <parent-task-id>-<n>-redo.json
├── runs/
│   └── <run-id>/
│       ├── run.json
│       ├── events.jsonl
│       └── worktree/
└── docs/
    └── (project docs from project.Docs)
```

### Epic

An Epic is a top-level feature request. It is never executed directly.
The orchestrator expands an Epic into a tree of Tasks.

Fields:
- `id`, `title`, `description`
- `project_id`
- `status`: open | in_progress | done | rejected
- `created_at`

### Task

A Task is a concrete step in the workflow.

Fields:
- `id`, `title`, `description`
- `type`: `research` | `qa_review` | `pm_plan` | `engineering` | `qa_verify` | `pm_consistency`
- `status`: open | in_progress | pending_review | approved | rejected | done
- `project_id`, `epic_id`, `parent_id`
- `dependencies`: list of task IDs that must be `done` before this task can start
- `specialization`: backend | frontend | devops | qa | pm | research | any
- `assigned_profile`: Hermes profile to run this task
- `verdict`: approve / reject (set by QA/PM agents)
- `verdict_reason`: free-form note
- `group`: stable ID grouping redo chains (the original task ID)
- `redo_index`: 0 for original, 1+ for redo tasks
- `created_at`, `started_at`, `ended_at`

### Task Type → Role mapping

| Task Type      | Required Agent Role | Gate output |
|----------------|---------------------|-------------|
| `research`     | researcher          | artifact: ТЗ / ADR / report |
| `qa_review`    | qa                  | verdict: approve / reject |
| `pm_plan`      | pm                  | artifact: decomposition JSON |
| `engineering`  | engineer / worker   | artifact: code / config changes |
| `qa_verify`    | qa                  | verdict: approve / reject |
| `pm_consistency`| pm                 | verdict: approve / reject |

The type determines the role. The role determines the Hermes profile.
No agent can receive a task of a mismatched type.

## Fixed transition graph

```yaml
transitions:
  research:
    approved:  qa_review
  qa_review:
    approved:  pm_plan
    rejected:  research        # creates a redo task in the same group
  pm_plan:
    approved:  engineering       # expands to N engineering tasks
  engineering:
    approved:  qa_verify
    rejected:  engineering       # redo the same engineering task
  qa_verify:
    approved:  pm_consistency
    rejected:  engineering       # redo
  pm_consistency:
    approved:  done
    rejected:  pm_plan           # or engineering, depending on reason
```

The orchestrator never invents a transition. It only follows the graph.

## Gate checks

Before a QA or PM agent is allowed to emit `approve`, the orchestrator runs
hard-coded checks that do not depend on the LLM:

```yaml
qa_verify.required_checks:
  - diff_exists:      runs/<run-id>/worktree differs from base branch
  - tests_pass:       if project defines a test command
  - lint_pass:        if project defines a lint command
  - review_note:      QA agent must list at least 3 checked items in metadata

pm_consistency.required_checks:
  - matches_tz:       diff is consistent with research artifact
  - no_merge_conflicts: worktree can be merged back to base
  - dependencies_done: all direct dependencies are approved/done
```

If a required check fails, the task stays `pending_review` and the agent must
fix the issue or the orchestrator creates a redo task.

## PM decomposition and parallel execution

The PM agent outputs a plan as JSON:

```json
{
  "tasks": [
    {
      "id": "eng-backend-auth",
      "type": "engineering",
      "specialization": "backend",
      "title": "JWT auth endpoint",
      "dependencies": []
    },
    {
      "id": "eng-db-users",
      "type": "engineering",
      "specialization": "backend",
      "title": "Add users table and migration",
      "dependencies": []
    },
    {
      "id": "eng-frontend-login",
      "type": "engineering",
      "specialization": "frontend",
      "title": "Login form and token storage",
      "dependencies": ["eng-backend-auth", "eng-db-users"]
    }
  ]
}
```

The orchestrator:
1. Validates acyclicity of the dependency graph.
2. Validates that every type is `engineering`.
3. Validates that every specialization is known to the project/team.
4. Creates tasks in `open` state.
5. Executes tasks layer by layer (topological sort).
   - Layer 0: backend + db in parallel.
   - Layer 1: frontend after both finish.

## Reject handling and redo chains

When a gate returns `reject`, the orchestrator:
1. Keeps the original task as `rejected`.
2. Creates a new task in the same `group` with `redo_index = previous + 1`.
3. Copies title/description/context and adds the rejection reason.
4. Returns the new task to the appropriate previous step.

Example chain:

```
tasks/
├── task-001.json           # research, approved
├── task-002.json           # qa_review, rejected (reason: missing security section)
├── task-002-1-redo.json    # research redo, approved
├── task-003.json           # qa_review of redo, approved
├── task-004.json           # pm_plan, approved
└── task-005.json           # engineering, ...
```

This keeps history visible and prevents an LLM from silently overwriting state.

## Agent invocation

Each task is executed by `cr run start --task <task-id>`.
The run package already handles git worktree, concurrency slots, and Hermes chat.
For workflow tasks the run passes extra metadata to the agent:
- task type and role,
- parent task artifacts,
- project docs,
- rejection reason (for redo tasks),
- required checks the gate expects.

The agent writes its verdict/artifact into `runs/<run-id>/metadata.json`.
The orchestrator reads that file and applies the deterministic transition.

## Orchestrator entry points

```bash
# start orchestrating an epic (foreground)
cr orchestrate --epic <epic-id>

# start orchestrating an epic (background)
cr orchestrate --epic <epic-id> --detach

# run a watchdog that watches all active epics
cr orchestrate --watch
```

## Safety rules

1. LLM cannot create task types outside the fixed list.
2. LLM cannot change the status of another task.
3. LLM cannot choose the next transition.
4. All approve/reject verdicts must pass hard gate checks.
5. Engineering tasks cannot start before their dependencies are `done`.
6. Redo tasks are always new records; old records are never mutated.

## Open questions

- Should `pm_consistency` reject go back to `pm_plan` or to a specific
  `engineering` redo? Proposed: reason starts with `pm:` → pm_plan, else
  engineering redo for the referenced task.
- Should QA checks include a `human_override` flag for the first few runs?
  Proposed: yes, via `workspace.yaml` `require_human_approval_for: [qa_verify]`.
